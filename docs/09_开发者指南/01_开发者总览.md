# 开发者总览

ZYArm 的开发链路从固件到上层应用是分层的。阅读代码前，先判断你要解决的问题属于哪一层，可以避免把简单问题改成跨模块联动。

## 先认识这些层

| 层级 | 它负责什么 | 常见修改 |
| --- | --- | --- |
| 固件 | 控制板上的真实运动、串口命令、状态上报、录制回放 | 新增 `[CMD]`、修改动作语义、调整状态字段 |
| 协议 | `[CMD]` 输入、`[STATUS]`/ACK 输出、单位和参数约定 | 增加参数、调整回包、补充错误语义 |
| SDK | 给 Python/C++ 调用者提供稳定 API | 封装新命令、统一单位、补测试 |
| ROS 2 | 机器人模型、控制器、MoveIt、Gazebo 和真机硬件接口 | 新 launch、新接口、新控制链 |
| Web | 浏览器中的模型加载、仿真展示和交互控制 | UI 控件、URDF 加载、拖拽逻辑 |
| LeRobot | 遥操、数据采集、回放和策略评估适配 | robot/teleoperator、camera 配置、动作映射 |
| 示例工具诊断 | 让用户学习、调试和压测 | 新示例、新工具、新压力测试 |

## 开发前先回答三个问题

1. 这个行为是在控制板上发生，还是在电脑程序里发生？
2. 改完后，用户会通过串口命令、SDK、ROS 2、Web 还是 LeRobot 使用它？
3. 需要怎样证明它可用：状态回包、脚本输出、截图、日志、视频还是数据集？

如果答案包含多个层级，先从最直接的模块章节开始，改完后再按 [测试发布与贡献检查](09_测试发布与贡献检查.md) 做跨层检查。

## 最小开发原则

- 能通过文档解释清楚的，不先改代码。
- 能在 SDK 封装的，不直接要求用户手写复杂串口命令。
- 能用工具脚本完成的，不把高级遥操流程塞进固件命令教程。
- 能用诊断脚本复现的问题，优先留下可重复的诊断入口。
- 改协议时同步更新文档和测试，不让上层调用者猜参数含义。
