# 示例工具与诊断开发

`software/examples/`、`software/tools/` 和 `software/diagnostics/` 都是脚本或程序，但定位不同。新增文件前先判断它是“教用户怎么做”、 “帮助日常操作”，还是“定位和压测问题”。

## 三类目录区别

| 目录 | 面向谁 | 适合放什么 |
| --- | --- | --- |
| `software/examples/` | 学习者、课程、应用开发者 | 抓取示例、相机示例、SDK 使用示例 |
| `software/tools/` | 日常使用者和维护者 | 读取设备信息、串口控制、主从遥操、硬件复位 |
| `software/diagnostics/` | 维护者和测试人员 | 压力测试、通信诊断、上下电循环、ACK/help 验证 |

源码入口：

- [software/examples/README.md](../../software/examples/README.md)
- [software/tools/README.md](../../software/tools/README.md)
- [software/diagnostics/README.md](../../software/diagnostics/README.md)

## 选择放在哪里

| 你要做的事 | 建议目录 |
| --- | --- |
| 展示如何用 SDK 完成一个任务 | `software/examples/` |
| 给用户一个日常可运行的操作工具 | `software/tools/` |
| 验证 100Hz、随机动作、上下电、通信稳定性 | `software/diagnostics/` |
| 临时排查个人机器问题 | 不建议直接提交，先整理成通用工具或诊断脚本 |

如果一个脚本同时能“教学”和“日常操作”，优先看默认使用者是谁。面向首次学习的，放 `examples`；面向频繁使用、需要参数化的，放 `tools`；面向维护和定位问题的，放 `diagnostics`。

## examples 开发建议

示例应让读者知道：

- 这个示例解决什么任务。
- 需要哪些硬件和外部 SDK。
- 如何安装依赖、编译和运行。
- 运行前机械臂应该处于什么姿态。
- 运行成功时能看到什么结果。

涉及摄像头、物块、桌面任务时，文档里必须保留照片、视野样张和参数占位。

## tools 开发建议

工具应偏向“日常可用”：

- 参数清楚，例如端口、波特率、设备角色。
- 错误信息能提示用户检查串口、供电和权限。
- 不把危险动作作为默认行为。
- 高级遥操优先通过脚本提供，不建议要求用户手工发送复杂高频固件命令。

当前常见工具包括 `get_arm_info.py`、`arm_serial_controller.py`、`filtered_master_slave_teleop.py`、`fast_io_teleop_pair.py` 和 Joy-Con 相关工具。

## diagnostics 开发建议

诊断脚本应偏向“可复现和可记录”：

- 输出测试条件，例如端口、频率、次数、持续时间。
- 输出失败原因和关键统计。
- 不默认执行大范围危险运动。
- 对压测脚本明确标注风险和停止方式。

当前诊断覆盖随机移动、固定移动、fast joint IO、上下电循环、录制动作、远程命令和 ACK/help 输出检查。

## 文档同步

新增示例、工具或诊断后，检查：

- 对应目录 README 是否列出新入口。
- [常用玩法与项目案例](../05_常用玩法/README.md) 是否需要引用。
- [校准维护与排障](../08_校准维护与排障/README.md) 是否需要引用诊断脚本。
- [科研与数据采集](../07_科研与数据采集/README.md) 是否受影响。
