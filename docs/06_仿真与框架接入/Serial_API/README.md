# Serial API 接入

Serial API 用于不依赖 ROS 2、SDK 或 Web，直接通过串口向固件发送 `[CMD]` 文本命令。它是最底层、最通用的接入方式，也是 Python SDK、ROS 2 真机桥接、Web 控制和 LeRobot 插件之下的通信基础。

本页只说明外部程序接入 Serial API 时应该关注什么。每条固件命令的详细含义、参数和安全注意事项，请优先阅读 [基础操作](../../04_基础操作/README.md)。

如果你要修改固件命令本身，请阅读 [固件开发](../../09_开发者指南/03_固件开发/README.md)。

## 适用场景

- 用串口软件或自写程序验证底层通信。
- 在自己的 Python、C++、Web 或其他上位机程序里直接发 `[CMD]`。
- 排查 SDK、ROS 2 或 Web 上层框架之前，先确认固件链路是否正常。
- 做一个最小依赖的调试工具。

不适合的场景：

- 需要运动规划时，优先使用 ROS 2 / MoveIt。
- 需要稳定 API 和单位转换时，优先使用 Python/C++ SDK。
- 需要遥操、数据采集和策略评估时，优先使用对应工具或 LeRobot 路径。

## 协议风格

当前固件采用文本命令格式，基本结构类似：

```text
[CMD][cmd_id][params...]
```

其中：

| 字段 | 含义 |
| --- | --- |
| `cmd_id` | 固件命令编号，例如状态查询、复位、IK、关节控制等 |
| `params` | 命令参数，不同 CMD 的数量和含义不同 |
| `ACK` | 固件对命令接收或执行过程的反馈 |
| `[STATUS]` | 固件输出的机械臂状态文本 |

命令执行过程中，通常会看到三类输出：

- `ACK_RECEIVED`
- 业务返回内容，例如 `[STATUS] ...`
- `ACK_COMPLETED`

外部程序接入时，不建议只判断串口有没有写入成功。更可靠的做法是同时处理 ACK、业务回包和超时。

## 推荐接入顺序

外部程序第一次接入时，建议从最小闭环开始：

1. 打开串口。
2. 发送状态查询 `CMD6`。
3. 确认能收到 `[STATUS]`。
4. 发送一次低风险小动作或复位命令。
5. 再次查询状态并记录。
6. 遇到异常时优先断电处理。

具体命令格式和动作建议见 [快速上手](../../02_快速上手/README.md) 和 [基础操作](../../04_基础操作/README.md)。

## 直接使用 Serial API 时要注意

- Serial API 不会自动做运动规划。
- Serial API 不会替你判断目标点是否安全。
- 直接使用固件角度时，要明确它不是 SDK/ROS 的用户角度表达。
- 连续控制前，先确认状态读取、复位和小动作稳定。
- 不要在未知姿态、未知零位或周围有障碍物时直接发送大动作。
- 外部程序应有超时、重试、日志和异常退出处理。

## 和其他框架的关系

| 上层能力 | 和 Serial API 的关系 |
| --- | --- |
| Python SDK | 封装 `[CMD]`、ACK、状态缓存和单位转换 |
| Web | 可以通过后续上位机服务间接调用串口能力 |
| ROS 2 真机桥接 | 在硬件接口中把 controller 命令转换为固件指令 |
| LeRobot | 通过 SDK 或工具链访问机械臂 |

如果你只是要写程序控制机械臂，通常先从 [Python 接入](../Python/README.md) 开始更省心。
